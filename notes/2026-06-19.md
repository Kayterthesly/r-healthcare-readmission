### 4. NEW TECHNICAL TERMS

| Term | My Simple Explanation |
|------|------------------------|
| `renv::install(prompt = FALSE)` | Installs packages without ever asking `[Y/n]` — the actual fix for the console prompt-eating bug, not a workaround |
| MinIO | Open-source, self-hosted object storage. Speaks the exact same S3 API as AWS S3 / Cloudflare R2 |
| `docker compose` | Defines and runs multi-container setups from one YAML file |
| `s3_use_ssl` | DuckDB setting controlling http (false) vs https (true) — MinIO local = false, real cloud = true |
| `s3_url_style='path'` | Tells S3-compatible clients to use bucket-in-path URLs; works across all providers |
| `register_cloud_tables()` | Creates DuckDB SQL views pointing at Parquet files in object storage, so later queries can say `FROM syn_person` instead of the full `s3://...` path |
| `.env` vs `.Renviron` | Docker Compose reads `.env` automatically; R reads `.Renviron` automatically — two different ecosystems, kept in sync manually |

### 5. WORKFLOW MAP

```
Console paste triggers install.packages() mid-block
  → renv shows [Y/n] prompt
    → next QUEUED LINE in the paste is consumed as the answer
      → if it's not y/n: "Unrecognized response", consumes ANOTHER line
        → entire blocks of real code silently never execute
FIX: renv::install(pkgs, prompt = FALSE) — never prompts, never eats lines
```

```

  ↓

  ↓

  ↓

  ↓

```

### 6. IMPORTANT RULES

```
Rule 1: Never paste a block containing install.packages() calls across
        multiple groups in one go — use renv::install(prompt = FALSE)
Rule 2: .Renviron and .env are both gitignored — real and fake secrets alike
Rule 3: Storage provider = 6 lines in .Renviron, zero lines of code
Rule 4: MinIO credentials in docker-compose.yml and .Renviron must match
        exactly, or authentication silently fails
Rule 5: DuckDB wants host:port (no protocol); paws wants the full URL
        with protocol — global_config.R handles this conversion
Rule 6: register_cloud_tables() returning 404s on empty bucket files is
        CORRECT behavior, not a failure — the files don't exist until
        Stage 1 generates and uploads them
```

### 8. MISTAKES / DEBUGGING

**The big one:** pasting all 10 package-install groups as one console block
caused renv's `[Y/n]` install prompts to silently eat several real code
blocks as failed answers — `duckdb`, `arrow`, and likely `RSQLite`, `paws`,
`tm`, `ellmer`, `plumber`, `shiny`/`shinydashboard` never actually got
installed on the first pass, even though no error appeared until the load
test. Root cause: a multi-line console paste is not atomic — if anything
mid-paste pauses for interactive input, everything queued after that point
becomes input to that prompt, not new code, until the prompt resolves.
Fixed by running an empirical `requireNamespace()` check across all 26
required packages, then installing only what was actually missing with
`renv::install(missing_pkgs, prompt = FALSE)` — confirmed clean on retry,
all 26 ✅ with no manual rescue needed.

**Second:** Cloudflare R2 signup got stuck on "Add R2 subscription to your
account." Confirmed via web search this is a real, recurring Cloudflare
billing bug reported by multiple users across many months — not something
done wrong. Pivoted to MinIO without losing any architectural progress,
because `global_config.R` was built provider-agnostic from the start.
